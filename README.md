# NetworkKB — Cisco / CCNP ENCOR Reference

**Current version: v3.3.0** — see [CHANGELOG.md](./CHANGELOG.md) for the
full version history. Versioning follows semver: MAJOR for overhauls,
MINOR for new features, PATCH for bug fixes — bump whichever applies
each time this project changes, and log it in the changelog.

Each interactive tool (Design Helper, DNS Builder, Flashcards, etc.) is
also versioned independently — see
[TOOL_VERSIONS.md](./TOOL_VERSIONS.md) for the current version and
patch history of each one specifically.

A self-contained Cisco networking reference and study tool: the OSI
model, DNS (with an interactive record builder), Cloud Platforms (with
an interactive shared responsibility model), ACLs, NAT, routing
(OSPF/EIGRP/BGP), switching, STP deep-dive, an interactive network
Design Helper (firewall/DMZ/WLC zones included), IPsec VPN, NGFW,
Cisco ISE, QoS, Wireless, global search, an in-app version history
browser, and 80 flashcards — all with links to official Cisco/IETF/
IEEE/NIST documentation.

This is the multi-file version, split out of the original single-HTML
build for easier maintenance:

## What's new — this round (major content + design tool expansion)

**New sections**
- **Fundamentals (OSI Model)** — all 7 layers with a visual stack diagram,
  full encapsulation/de-encapsulation walkthrough with a worked example,
  layer-by-layer deep dive, OSI vs TCP/IP comparison, and troubleshooting
  mapped to each layer (the bottom-up method).
- **DNS** — how resolution actually works (recursive vs iterative, with a
  diagram), every major record type with real examples, and an
  **interactive record builder tool** — pick A/AAAA/CNAME/MX/TXT/NS/PTR/
  SRV/SOA and the form fields change to match what that record actually
  needs, with a one-click example and live zone-file output.
- **Security (merged)** — IPsec VPN content moved here and joined by two
  new reference-only sections: **NGFW** (what makes a firewall
  "next-gen," core features like App-ID/DPI/IPS/TLS inspection) and
  **Cisco ISE** (identity-based access control, 802.1X/MAB, TrustSec/SGT
  policy enforcement). No configuration tools for these two — reference
  content only, as requested.
- **18 new flashcards** across OSI, DNS, and Security (74 total, up from
  56), with matching filter buttons in the Flashcards tool.

**Design Helper — zones, firewall, DMZ, WLC**
- The topology diagram now draws a **strict visual split**: bordered
  External/WAN, DMZ, and Internal zones, matching how a real perimeter
  is actually segmented.
- New optional appliances: a **perimeter firewall** (HA pair or single,
  following the same redundancy setting as everything else), a **DMZ**
  with an example public-facing server (only offered when a firewall is
  present — a DMZ without a firewall isn't really a DMZ), and a
  **Wireless LAN Controller** (only offered when wireless is requested).
  All three are optional toggles in the requirements form.
- Every new device type is clickable, same as before — the firewall
  shows a skeleton perimeter policy, the DMZ server shows its isolation
  rationale, the WLC shows its CAPWAP/SSID starter config.
- A few modern icon touches: a real cloud shape for the External zone,
  small glyphs (shield, wifi arcs, server rack) on the firewall/WLC/DMZ
  nodes instead of plain text-only boxes.
- **Layout overhaul**: the layer-toggle legend moved from a horizontal
  bar above the diagram to a vertical column beside it, so the diagram
  itself gets significantly more width. The input form and the four
  detail cards (topology/devices/VLANs/starter config) are now
  collapsible — the form auto-collapses right after you generate a
  design, so the diagram stays the dominant thing on screen instead of
  competing with four full data tables for space.

**Navigation**
- Merged the standalone VPN section into Security (IPsec is a security
  topic anyway) once the growing section count started actually
  overflowing the top nav at standard widths — confirmed with a real
  measurement before touching anything, not a guess.

## Why the split

- **CSS/JS/HTML separated** — easier to edit one thing without scrolling
  through a 300KB file, and your editor's syntax highlighting/linting
  actually works properly per file type.
- **Flashcards as JSON** — content lives in `data/flashcards.json`, not
  buried in a JS array. Add/edit cards without touching any code.
- **One canonical nav.js** — the original single-file build had picked up
  two competing `goPage()` functions over several editing passes (a real
  bug we chased down earlier). Splitting forced a cleanup — there's now
  exactly one definition.
- **PHP backend is optional** — everything except Save/Load Design works
  with zero backend, even opened directly as a `file://` page. Only the
  Design Helper's save/reload feature needs a server, because it needs
  somewhere to persist your saved designs (SQLite file in `data/`).

## Running it locally (test before deploying to your homelab)

You need PHP installed locally (PHP 7.4+ recommended; the SQLite
extension `pdo_sqlite` must be enabled — it's on by default in nearly
every distro's PHP package).

```bash
cd NetworkKB-v2
php -S localhost:8000
```

Then open **http://localhost:8000/index.html** in a browser.

That single command serves the static files (HTML/CSS/JS/JSON) *and*
runs the PHP API endpoints — no separate web server needed for local
testing.

### Verifying it works

1. **Static content** — Home, ACL, NAT, Routing, Switching, Design,
   VPN, QoS, Wireless, Quick Ref should all navigate normally.
2. **Flashcards** — click Flashcards in the top nav; you should see
   "1 / 56" and a question. (If this is blank, you likely opened
   `index.html` directly as a `file://` page instead of through
   `php -S` — browsers block `fetch()` of local JSON files over
   `file://` for security reasons. This is the one feature that
   requires being served over http.)
3. **Design Helper** — go to Design → 🧮 Design Helper, fill in the
   form, click "Generate design". You should get a visual topology
   diagram (click a device icon to see its config in a popup), the
   layer checkboxes above it should show/hide parts of the diagram
   instantly, plus device counts, a VLAN table, and a starter config.
4. **Save/Load** — click "💾 Save design", you should get an alert
   confirming it saved with an ID. Click "📂 Load saved design" to see
   it listed. If you get an error here instead, see Troubleshooting
   below.

### If PHP isn't installed yet

```bash
# Debian/Ubuntu
sudo apt install php php-sqlite3

# Fedora/RHEL
sudo dnf install php php-pdo

# macOS (Homebrew)
brew install php
```

## Moving to your homelab (Docker) — step by step

This folder now includes a `Dockerfile` and `docker-compose.yml` built
specifically for permanent hosting. Here's what each step does and why,
so you're learning the pattern, not just running commands.

### The pieces, at a glance

- **Dockerfile** — the recipe for building an image: start from PHP +
  Apache, install the one missing piece (SQLite support), copy the app
  in, fix a permissions issue before it can bite you. You build this
  once (and again whenever the app's code changes).
- **docker-compose.yml** — how to actually *run* that image: which port
  to expose, which folder on your host to treat as permanent storage,
  and what to do if it crashes or the host reboots. You run this every
  time you start/stop/update the service.
- **./data/** (a folder, created the first time you run it) — the one
  thing that needs to survive container rebuilds: your saved designs.

### Step 1 — confirm Docker is ready on the host

SSH into whichever homelab box will run this (a VM or LXC with Docker
installed — Proxmox users typically use a small Ubuntu/Debian VM or an
unprivileged LXC for this):

```bash
docker --version
docker compose version
```

Both should print a version number. If either command isn't found,
install Docker first (e.g. `curl -fsSL https://get.docker.com | sh` on
Debian/Ubuntu), then come back here.

### Step 2 — copy the project onto that host

From wherever you unzipped `NetworkKB-v2`, copy the whole folder to the
Docker host — `scp` is the simplest way if you're doing this once:

```bash
scp -r NetworkKB-v2 you@your-homelab-host:/home/you/networkkb
```

(If you'll be editing content over time, cloning from a private git repo
instead of `scp` makes future updates a `git pull` — up to you.)

### Step 3 — look at what's already in the folder

```bash
cd /home/you/networkkb
cat Dockerfile
cat docker-compose.yml
```

Both files have comments explaining every line — read through them once;
that's most of the "how Docker works" learning right there. The short
version: `Dockerfile` builds the image, `docker-compose.yml` runs it.

### Step 4 — build and start it

```bash
docker compose up -d --build
```

- `up` — create and start the service defined in `docker-compose.yml`
- `-d` — "detached": run in the background instead of tying up your
  terminal
- `--build` — build the image from the Dockerfile first (needed the
  first time, and again any time you change the app's code)

First run will take a minute or two (downloading the base PHP+Apache
image, installing the SQLite extension). Subsequent runs are fast since
Docker caches the layers that haven't changed.

### Step 5 — verify it's actually running

```bash
docker compose ps
```

You should see the `networkkb` service listed as `running`. Then browse
to:

```
http://<your-homelab-ip>:8080/index.html
```

Check the same three things from the local-testing section earlier:
flashcards load (confirms PHP/Apache is serving `data/flashcards.json`
correctly), the Design Helper generates a diagram, and Save Design
works without an error.

If something's wrong, check the container's logs:

```bash
docker compose logs -f
```

(Ctrl+C to stop following.)

### Step 6 — confirm data actually persists

This is the step that's easy to skip and annoying to find out later
that you skipped. Save a test design, then:

```bash
docker compose down     # stops AND removes the container
docker compose up -d    # recreates it from scratch
```

Reload the page and check "Load saved design" — your test design should
still be there. That's the `./data:/var/www/html/data` volume mount
doing its job: the SQLite file lives on the host's disk, not inside the
disposable container.

### Step 7 — updating the app later

When you come back to add more content or fix something:

```bash
# edit files as usual, then:
docker compose up -d --build
```

Compose rebuilds the image with your changes and swaps the running
container for the new one — the `./data` folder (and everything in it)
is untouched, because it was never inside the container to begin with.

### Optional — a friendly hostname instead of `ip:8080`

Right now you're hitting it by IP and port. To get something like
`http://networkkb.home.lab` instead, put a reverse proxy in front of it
— any of these work well in a homelab:

- **Nginx Proxy Manager** (its own Docker container with a web UI —
  probably the easiest if you don't already have a proxy)
- **Traefik** (more powerful, more config, integrates natively with
  Docker Compose via labels)
- **pfSense + HAProxy package**, if you'd rather keep it inside pfSense
  since that's already part of your stack

Whichever you pick, it just needs to forward `networkkb.home.lab` (or
whatever name you choose) to `<homelab-ip>:8080`. Add a DNS entry for
that hostname in Pi-hole so it resolves on your LAN.

### Backups

The entire "database" is one file: `./data/networkkb.sqlite`. Back up
that file (or the whole `./data` folder) however you already back up
this host — it's small, portable, and doesn't need the container running
to copy.



## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| Flashcards page is blank | Opened `index.html` directly (`file://`) | Serve via `php -S` or Docker instead |
| "Could not reach the save API" alert | Same as above, or PHP not running | Confirm `php -S` is running and you're browsing `http://localhost:8000/...` not a `file://` path |
| Save works locally but not after moving to Docker | `pdo_sqlite` extension not installed in the container | Run the `docker-php-ext-install pdo_sqlite` step above |
| 500 error from an `api/*.php` call | Check the container/server's PHP error log | `docker logs networkkb` (Option A/B) or your Apache/PHP error log |
| Saved designs "disappeared" after a Docker rebuild | Volume wasn't mounted, so the sqlite file lived inside the throwaway container | Make sure `-v /path/to/NetworkKB-v2:/var/www/html` (or the compose `volumes:` line) is in place *before* saving anything you care about |

## Editing content

- **Changing a tool's behavior?** Bump its version — update the
  `<span class="tool-version-tag">` next to its title in `index.html`
  and add an entry to [TOOL_VERSIONS.md](./TOOL_VERSIONS.md). PATCH for
  a bug fix, MINOR for a real capability added, MAJOR for a rebuild.
  This is what makes "what changed in the DNS builder" answerable
  without reading the whole product changelog.
- **Add/edit flashcards** — edit `data/flashcards.json` directly. Each
  entry is `["question", "answer", "category"]`. No JS knowledge needed.
- **Add a new reference panel** — copy an existing `<div class="panel"
  data-parent="...">...</div>` block in `index.html`, give it a new
  `id`, and add a matching sidebar `<div class="sb-item" data-sub="...">`
  entry.
- **Tweak the design helper's math** — all of the sizing/topology logic
  lives in `computeDesign()` in `js/design-helper.js`, in one place, with
  the rest of the file just doing rendering and API calls.
- **After adding or editing any panel** — regenerate the search index so
  the new/changed content is actually findable via `/`:
  ```bash
  python3 scripts/build_search_index.py
  ```
  It's a static JSON file, not built automatically on page load, so this
  is a manual step you run once after editing `index.html` — the search
  overlay itself needs no server-side anything, it just fetches whatever
  `data/search-index.json` currently contains.
- **After editing CHANGELOG.md** — regenerate the in-app version history
  viewer the same way:
  ```bash
  python3 scripts/build_changelog_json.py
  ```
  Write new bullets as one long line each, not manually hard-wrapped —
  see the comment at the top of that script for why (a wrapped bullet
  that breaks mid-word or mid slash-chain parses with a stray space).
